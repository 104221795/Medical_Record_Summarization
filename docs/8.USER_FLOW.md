# User Flow — Hệ thống tóm tắt bệnh án tích hợp HIS/EMR

## 1. Mục tiêu tài liệu

Tài liệu này mô tả **user flow / workflow** cho hệ thống **Medical Record Summarization System tích hợp HIS/EMR**.

Mục tiêu là làm rõ:

- Ai là người sử dụng hệ thống.
- Mỗi role có quyền gì và thao tác gì.
- Luồng người dùng từ lúc mở hồ sơ bệnh nhân đến khi tạo, kiểm tra, chỉnh sửa và phê duyệt summary.
- Luồng xử lý citation, unsupported claim và hallucination warning.
- Luồng admin theo dõi chất lượng hệ thống.
- Luồng audit và monitoring để đảm bảo truy vết.

Nguyên tắc thiết kế quan trọng:

> AI-generated summary luôn là bản nháp. Bác sĩ phải kiểm tra, chỉnh sửa và phê duyệt trước khi nội dung được xem là usable trong workflow lâm sàng.

# 3. End-to-End User Flow Overview

## 3.1 Main product flow

```mermaid
flowchart TD
    A[User Login] --> B{Role Check}
    B -->|Doctor| C[Patient List]
    B -->|Nurse| C
    B -->|Clinical Admin| D[Quality Dashboard]
    B -->|IT Admin| E[Admin Console]
    B -->|AI Safety Reviewer| F[Safety Review Dashboard]
    B -->|Auditor| G[Audit Log Viewer]

    C --> H[Patient Detail]
    H --> I[Clinical Documents / Encounter Data]
    H --> J[AI Summary Workspace]
    J --> K[Generate Summary]
    K --> L[Draft Summary with Citations]
    L --> M[Safety Check Result]
    M --> N{Doctor Review}
    N -->|Approve| O[Approved Summary]
    N -->|Edit| P[Edited Summary]
    P --> O
    N -->|Reject| Q[Rejected Summary]
    N -->|Regenerate| K

    O --> R[Export / Draft Write-back]
    O --> S[Audit Log]
    Q --> S
    P --> S
```

## 3.2 Core flow statement

```text
User logs in
→ Opens patient profile
→ Reviews clinical data
→ Generates AI summary
→ Checks citations and safety warnings
→ Edits if needed
→ Approves or rejects
→ System stores version, review action and audit log
```

# 4. Doctor User Flow

Doctor là role quan trọng nhất trong MVP vì bác sĩ là người trực tiếp đánh giá tính hữu ích và an toàn của AI summary.

---

## 4.1 Doctor happy path — Generate and approve summary

```mermaid
sequenceDiagram
    participant Doctor
    participant UI
    participant API
    participant SummaryService
    participant SafetyService
    participant DB

    Doctor->>UI: Login
    UI->>API: Authenticate user
    API-->>UI: Return doctor session

    Doctor->>UI: Open patient list
    UI->>API: Get assigned patients
    API-->>UI: Return patient list

    Doctor->>UI: Select patient
    UI->>API: Get patient detail and encounters
    API-->>UI: Return patient context

    Doctor->>UI: Click Generate AI Summary
    UI->>SummaryService: Request summary generation
    SummaryService->>DB: Retrieve clinical data and document chunks
    SummaryService->>SummaryService: Generate citation-grounded summary
    SummaryService->>SafetyService: Verify claims and citations
    SafetyService-->>SummaryService: Return safety result
    SummaryService->>DB: Save draft summary, claims, citations
    SummaryService-->>UI: Return draft summary

    Doctor->>UI: Review summary and citations
    Doctor->>UI: Approve summary
    UI->>API: Submit approval
    API->>DB: Save approved version and audit log
    API-->>UI: Approval successful
```

---

## 4.2 Doctor detailed steps

| Step | Doctor action | System response |
|---|---|---|
| 1 | Login | Xác thực user và role |
| 2 | Mở danh sách bệnh nhân | Hiển thị bệnh nhân được phân quyền |
| 3 | Chọn bệnh nhân | Hiển thị patient detail, encounters, documents |
| 4 | Chọn loại summary | Patient Snapshot / Clinical Timeline / Discharge Draft |
| 5 | Click Generate | Hệ thống tạo draft summary |
| 6 | Xem bản summary | Hiển thị summary theo section |
| 7 | Click citation | Mở source note/lab/report gốc |
| 8 | Kiểm tra safety warning | Hiển thị unsupported/conflicting claims nếu có |
| 9 | Chỉnh sửa nếu cần | Lưu edited version |
| 10 | Approve hoặc Reject | Cập nhật trạng thái summary và audit log |

---

## 4.3 Doctor flow — Edit summary before approval

```mermaid
flowchart TD
    A[Draft Summary Generated] --> B[Doctor Reviews Summary]
    B --> C{Is summary acceptable?}
    C -->|Yes| D[Approve Summary]
    C -->|Mostly yes but needs correction| E[Edit Summary]
    E --> F[Save Edited Version]
    F --> G[Review Final Version]
    G --> H{Approve edited version?}
    H -->|Yes| D
    H -->|No| I[Reject Summary]
    C -->|No| I
    D --> J[Status = Approved]
    I --> K[Status = Rejected]
    J --> L[Audit Log Created]
    K --> L
```

## 4.4 Edit behavior

Khi bác sĩ chỉnh sửa summary:

- Hệ thống phải lưu bản gốc do AI tạo.
- Hệ thống phải lưu bản edited text.
- Hệ thống nên tính edit distance nếu có.
- Hệ thống phải ghi audit log.
- Hệ thống phải giữ citation mapping nếu claim không đổi.
- Nếu bác sĩ thêm claim mới không có citation, hệ thống nên flag là “clinician-added / citation not available”.

---

## 4.5 Doctor flow — Reject summary

```mermaid
flowchart TD
    A[Doctor Reviews Draft Summary] --> B{Problem Found?}
    B -->|Unsupported claim| C[Reject with reason: Unsupported claim]
    B -->|Wrong citation| D[Reject with reason: Citation issue]
    B -->|Missing key information| E[Reject with reason: Missing information]
    B -->|Unsafe wording| F[Reject with reason: Safety concern]
    B -->|Too generic| G[Reject with reason: Low usefulness]
    C --> H[Status = Rejected]
    D --> H
    E --> H
    F --> H
    G --> H
    H --> I[Audit Log Created]
    H --> J[Feedback Sent to Quality Dashboard]
```

## 4.6 Reject reason taxonomy

| Reject reason | Meaning |
|---|---|
| `unsupported_claim` | AI đưa ra thông tin không có bằng chứng |
| `wrong_citation` | Citation không hỗ trợ claim |
| `missing_critical_info` | Thiếu thông tin quan trọng |
| `incorrect_clinical_fact` | Sai thông tin lâm sàng |
| `conflicting_evidence` | Nguồn dữ liệu mâu thuẫn |
| `poor_readability` | Diễn đạt khó hiểu |
| `too_generic` | Summary quá chung chung |
| `unsafe_output` | Có rủi ro gây hiểu nhầm hoặc nguy hiểm |
| `other` | Khác |

# 6. Summary Status Flow

## 6.1 Summary state machine

```mermaid
stateDiagram-v2
    [*] --> Draft: AI generated
    Draft --> UnderReview: Doctor opens review
    UnderReview --> Edited: Doctor edits
    UnderReview --> Approved: Doctor approves
    UnderReview --> Rejected: Doctor rejects
    Edited --> UnderReview: Save edit
    Edited --> Approved: Doctor approves
    Edited --> Rejected: Doctor rejects
    Rejected --> Draft: Regenerate
    Approved --> Archived: New version generated
    Archived --> [*]
```

---

## 6.2 Status definitions

| Status | Meaning | Editable? | Official? |
|---|---|---:|---:|
| `draft` | AI tạo, chưa review | Yes | No |
| `under_review` | Bác sĩ đang xem | Yes | No |
| `edited` | Đã được chỉnh sửa | Yes | No |
| `approved` | Đã được bác sĩ phê duyệt | Limited | Yes/Usable |
| `rejected` | Bị từ chối | No | No |
| `archived` | Phiên bản cũ | No | No |

# 16. MVP User Flow Summary

MVP user flow nên tập trung vào 5 luồng chính:

```text
1. Doctor generates citation-grounded summary
2. Doctor verifies citation and source evidence
3. Doctor edits, approves or rejects summary
4. IT Admin imports mock/FHIR-like clinical data
5. Clinical Admin / Safety Reviewer monitors quality and flagged claims
```

Các flow khác như patient-facing app, voice assistant, automated prescription, treatment recommendation hoặc direct EMR write-back không nằm trong MVP.
