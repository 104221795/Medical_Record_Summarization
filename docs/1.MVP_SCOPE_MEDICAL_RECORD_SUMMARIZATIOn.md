# MVP Scope — Hệ thống tóm tắt bệnh án tích hợp HIS/EMR

## 1. Mục tiêu MVP

MVP của hệ thống tập trung vào việc xây dựng một **Clinical Record Summarization Assistant** có khả năng tóm tắt bệnh án từ dữ liệu HIS/EMR ở mức an toàn, có dẫn nguồn và có bác sĩ kiểm duyệt.

Mục tiêu của MVP không phải là chẩn đoán, gợi ý điều trị hay tự động ra quyết định lâm sàng. Thay vào đó, MVP nhằm chứng minh rằng hệ thống có thể:

- Trích xuất dữ liệu bệnh án từ nguồn EMR/HIS hoặc dữ liệu mô phỏng.
- Tóm tắt thông tin bệnh nhân theo cấu trúc lâm sàng rõ ràng.
- Gắn citation cho từng thông tin quan trọng trong bản tóm tắt.
- Phát hiện hoặc hạn chế các claim không có bằng chứng.
- Cho phép bác sĩ xem, chỉnh sửa và phê duyệt bản tóm tắt.
- Lưu lại lịch sử tạo, chỉnh sửa và phê duyệt để phục vụ audit.

## 2. MVP In Scope

### 2.1 Data Input

MVP sẽ hỗ trợ các nguồn dữ liệu cơ bản sau:

| Data type | Description | MVP priority |
|---|---|---|
| Patient demographic | Thông tin cơ bản: tuổi, giới tính, mã bệnh nhân ẩn danh | Must-have |
| Encounter data | Lần khám/đợt điều trị, ngày nhập viện, ngày ra viện | Must-have |
| Clinical notes | Ghi chú bác sĩ, progress note, admission note, discharge note | Must-have |
| Diagnosis / Problem list | Chẩn đoán hiện tại, bệnh nền, vấn đề đang theo dõi | Must-have |
| Medication data | Thuốc đang dùng, thuốc đã ngưng, thay đổi thuốc | Should-have |
| Lab results | Một số chỉ số xét nghiệm quan trọng | Should-have |
| Vital signs | Huyết áp, mạch, nhiệt độ, SpO2 nếu có | Should-have |
| Imaging report text | Báo cáo hình ảnh dạng text, không phân tích ảnh trực tiếp | Could-have |

MVP **không cần tích hợp PACS hoặc phân tích ảnh y tế trực tiếp**. Nếu có dữ liệu hình ảnh, hệ thống chỉ xử lý phần **radiology report dạng văn bản**.

### 2.2 Summary Generation

MVP cần tạo được các loại summary sau:

| Summary type | Description | Priority |
|---|---|---|
| Patient Snapshot | Tóm tắt nhanh tình trạng bệnh nhân | Must-have |
| Active Problem Summary | Tóm tắt các vấn đề/chẩn đoán đang được theo dõi | Must-have |
| Recent Clinical Timeline | Tóm tắt diễn biến chính theo thời gian | Must-have |
| Medication Summary | Tóm tắt thuốc hiện tại và thay đổi thuốc | Should-have |
| Lab Highlight Summary | Tóm tắt xét nghiệm bất thường hoặc đáng chú ý | Should-have |
| Discharge Summary Draft | Bản nháp tóm tắt ra viện | Should-have |

Cấu trúc output nên cố định theo template:

```text
1. Patient Snapshot
2. Active Problems
3. Recent Clinical Course
4. Medications
5. Labs and Imaging Highlights
6. Pending Issues / Follow-up
7. Notes Requiring Clinician Review
```

### 2.3 Citation-based Summary

Đây là tính năng bắt buộc trong MVP.

Mỗi thông tin y khoa quan trọng trong bản tóm tắt phải có nguồn dẫn về dữ liệu gốc.

Ví dụ:

```text
Patient has a history of type 2 diabetes. [Source: Progress Note, 2026-05-20]
Creatinine increased from 1.1 to 1.8 mg/dL. [Source: Lab Result, 2026-05-21]
Metformin was discontinued during admission. [Source: Medication Note, 2026-05-22]
```

MVP cần hỗ trợ:

| Feature | Description | Priority |
|---|---|---|
| Source linking | Mỗi claim có link hoặc reference đến note/result gốc | Must-have |
| Source metadata | Hiển thị loại tài liệu, ngày giờ, nguồn hệ thống | Must-have |
| Claim-source mapping | Lưu mapping giữa câu summary và nguồn bằng chứng | Must-have |
| Citation coverage score | Tính tỷ lệ claim có citation | Should-have |

Rule quan trọng:

> Nếu một claim y khoa không có nguồn bằng chứng, hệ thống không được trình bày claim đó như thông tin chắc chắn. Claim đó phải bị loại bỏ hoặc đưa vào nhóm “Needs Clinician Review”.

### 2.4 Hallucination Mitigation

MVP cần có safety layer để giảm rủi ro AI bịa thông tin.

Các cơ chế trong scope:

| Safety feature | Description | Priority |
|---|---|---|
| Grounded generation | Model chỉ được tóm tắt dựa trên context được retrieve | Must-have |
| Citation enforcement | Claim quan trọng bắt buộc có citation | Must-have |
| Unsupported claim detection | Phát hiện câu không có bằng chứng trong nguồn | Must-have |
| Conflict flagging | Đánh dấu khi dữ liệu nguồn có mâu thuẫn | Should-have |
| Clinician review required | Bản tóm tắt luôn ở trạng thái draft trước khi bác sĩ duyệt | Must-have |
| AI disclaimer | Hiển thị rõ đây là nội dung AI hỗ trợ, chưa phải kết luận chính thức | Must-have |

MVP không cần xây một mô hình hallucination detection phức tạp ngay từ đầu. Có thể bắt đầu bằng rule-based + LLM-based verification:

```text
Generated summary → Split into claims → Check each claim against retrieved evidence → Mark as Supported / Unsupported / Conflicting
```

### 2.5 Human-in-the-loop Review

MVP bắt buộc có quy trình bác sĩ duyệt.

Workflow tối thiểu:

```text
AI generates draft summary
→ Doctor reviews summary
→ Doctor checks citations
→ Doctor edits if needed
→ Doctor approves or rejects
→ System stores final version and audit log
```

Các trạng thái của summary:

| Status | Meaning |
|---|---|
| Draft | AI vừa tạo, chưa có bác sĩ duyệt |
| Under Review | Bác sĩ đang xem/chỉnh sửa |
| Approved | Đã được bác sĩ phê duyệt |
| Rejected | Bác sĩ từ chối sử dụng |
| Archived | Phiên bản cũ, không còn dùng |

---

### 2.6 EMR/HIS Integration

Ở MVP, không cần tích hợp sâu ngay với bệnh viện thật. Nên chia thành 2 mức:

#### MVP Level 1 — Demo/Sandbox

Hệ thống có thể nhập dữ liệu từ:

- File JSON giả lập theo cấu trúc FHIR.
- CSV export từ HIS/EMR.
- Clinical notes dạng `.txt` hoặc `.docx`.
- Dataset public/de-identified như MIMIC-IV-Note nếu có quyền truy cập.

#### MVP Level 2 — Integration-ready

Thiết kế hệ thống phải sẵn sàng mapping sang các FHIR resources:

| Clinical concept | FHIR resource |
|---|---|
| Patient information | Patient |
| Encounter / admission | Encounter |
| Diagnosis / problem | Condition |
| Lab / vital result | Observation |
| Medication order | MedicationRequest |
| Imaging report | DiagnosticReport |
| Clinical note | DocumentReference / Composition |

MVP nên ưu tiên **FHIR-compatible data model**, kể cả khi chưa kết nối trực tiếp với HIS thật.

### 2.7 Admin & Monitoring Dashboard

MVP nên có dashboard đơn giản cho admin/product owner/clinical lead.

Các chỉ số cần hiển thị:

| Metric | Description |
|---|---|
| Number of summaries generated | Số bản tóm tắt đã tạo |
| Approval rate | Tỷ lệ được bác sĩ phê duyệt |
| Rejection rate | Tỷ lệ bị từ chối |
| Average edit distance | Mức độ bác sĩ phải sửa nhiều hay ít |
| Citation coverage | Tỷ lệ claim có citation |
| Unsupported claim count | Số claim không có bằng chứng |
| Average generation time | Thời gian tạo summary |
| User activity | Ai tạo/xem/duyệt summary |

## 3. MVP Out of Scope

Các phần sau **không nằm trong MVP**:

| Feature | Reason |
|---|---|
| AI diagnosis recommendation | Rủi ro clinical decision support cao |
| Treatment recommendation | Dễ vượt phạm vi documentation assistant |
| Automatic prescription | Rủi ro pháp lý và an toàn cao |
| Autonomous discharge approval | Bắt buộc cần bác sĩ duyệt |
| Medical image diagnosis | Cần dataset, validation và regulatory phức tạp |
| Full voice ambient scribe | Scope lớn, nên để phase sau |
| Patient-facing chatbot | Rủi ro hiểu sai thông tin y khoa |
| Billing/coding automation | Có thể làm sau khi summary ổn định |
| Direct write-back vào EMR chính thức | MVP chỉ nên write-back dạng draft hoặc export |
| Fine-tuning trên dữ liệu thật | Chỉ làm khi có phê duyệt dữ liệu và governance rõ |

## 6. MVP Success Criteria

MVP được xem là thành công nếu đạt các tiêu chí sau:

| Area | Success metric | Target |
|---|---|---|
| Functional completeness | Use case chính chạy end-to-end | ≥ 90% |
| Citation quality | Clinical claims có citation | ≥ 90–95% |
| Safety | Unsupported claim rate sau filter | < 5% trong MVP |
| Usability | Bác sĩ hiểu được summary và citation | ≥ 4/5 rating |
| Efficiency | Giảm thời gian đọc hồ sơ | 20–40% trong test scenario |
| Review workflow | Bác sĩ có thể edit/approve/reject | 100% supported |
| Traceability | Mọi summary có version + audit log | 100% |
| Performance | Patient snapshot generation time | < 15 giây với record ngắn/trung bình |

## 7. Recommended MVP Build Order

Thứ tự build nên là:

```text
1. Data ingestion from mock/FHIR-like JSON
2. Clinical note chunking and indexing
3. RAG-based retrieval
4. Structured summary generation
5. Citation mapping
6. Unsupported claim checker
7. Doctor review UI
8. Approval/rejection workflow
9. Audit log
10. Monitoring dashboard
```

Không nên bắt đầu bằng Whisper, ViT hoặc Edge AI. Các công nghệ đó hữu ích nhưng không phải core của MVP. Core MVP phải là:

```text
EMR data → Retrieval → Citation-grounded Summary → Safety Check → Doctor Review
```

---

## 8. MVP Boundary Statement

> MVP của hệ thống tập trung vào việc tạo bản tóm tắt bệnh án có cấu trúc, có dẫn nguồn và có bác sĩ kiểm duyệt từ dữ liệu HIS/EMR hoặc dữ liệu mô phỏng theo chuẩn FHIR. Hệ thống không đưa ra chẩn đoán, không khuyến nghị điều trị, không kê đơn và không tự động ghi nhận nội dung vào hồ sơ bệnh án chính thức khi chưa có xác nhận của bác sĩ.
