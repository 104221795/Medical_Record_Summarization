# API Specification — Medical Record Summarization System

## 1. Mục tiêu tài liệu

Tài liệu này mô tả **API Specification** cho hệ thống **Medical Record Summarization tích hợp HIS/EMR**.

API được thiết kế để hỗ trợ các nhóm chức năng chính:

- Quản lý user, role và phân quyền.
- Truy xuất patient, encounter và clinical documents.
- Import dữ liệu bệnh án từ file/API/FHIR-like JSON.
- Tạo bản tóm tắt bệnh án bằng AI.
- Gắn citation cho từng claim trong summary.
- Kiểm tra hallucination, unsupported claim và conflicting evidence.
- Cho phép bác sĩ review, chỉnh sửa, approve hoặc reject summary.
- Ghi audit log và theo dõi model run.
- Chuẩn bị khả năng tích hợp FHIR/SMART trong production.

API này phù hợp cho MVP dùng **FastAPI + PostgreSQL + Vector DB**, đồng thời có thể mở rộng sang kiến trúc production.

---

# 2. API Design Principles

| Principle | Description |
|---|---|
| RESTful-first | API dùng REST endpoint rõ ràng, dễ tích hợp frontend |
| FHIR-compatible | Dữ liệu patient, encounter, observation, document nên mapping được sang FHIR |
| Role-based access | Mỗi endpoint cần kiểm tra quyền theo role |
| Audit by default | Các hành động nhạy cảm phải ghi audit log |
| Draft-first AI output | Summary AI tạo ra luôn là draft trước khi bác sĩ approve |
| Citation-first | Claim y khoa quan trọng phải có citation |
| Versioned API | Dùng `/api/v1` để dễ mở rộng |
| Structured error response | Error trả về theo schema thống nhất |
| Idempotent ingestion | Import dữ liệu nên tránh tạo duplicate |
| Safe write-back | Không ghi summary vào EMR chính thức nếu chưa có review |

Khi thay đổi breaking change, tạo version mới:

```text

```

# 4. Authentication and Authorization

## 4.1 MVP Authentication

Ở MVP có thể dùng JWT-based login.

```http
Authorization: Bearer <access_token>
```

## 4.2 Production Authentication

Ở production, nên dùng:

- Hospital SSO.
- OAuth2/OpenID Connect.
- SMART App Launch nếu app được nhúng trong EHR.
- Service-to-service token cho backend integration.

## 4.3 Role codes

```text
doctor
nurse
clinical_admin
it_admin
ai_safety_reviewer
auditor
```

## 4.4 Common authorization rules

| Action | Required role |
|---|---|
| Generate summary | doctor, clinical_admin |
| Approve/reject summary | doctor |
| Import data | it_admin |
| View quality dashboard | clinical_admin, ai_safety_reviewer, it_admin |
| View audit logs | clinical_admin, it_admin, auditor |
| Configure source systems | it_admin |

# 12. Summary API

## 12.1 Generate summary

```http
POST /api/v1/patients/{patient_id}/summaries/generate
```

### Request

```json
{
  "encounter_id": "enc_001",
  "summary_type": "patient_snapshot",
  "language": "vi",
  "date_range": {
    "from": "2026-05-20",
    "to": "2026-05-27"
  },
  "options": {
    "require_citations": true,
    "include_safety_check": true,
    "retrieval_top_k": 12
  }
}
```

### Response

```json
{
  "success": true,
  "data": {
    "summary_id": "sum_001",
    "status": "draft",
    "summary_type": "patient_snapshot",
    "summary_language": "vi",
    "citation_coverage": 0.95,
    "unsupported_claim_count": 1,
    "conflict_count": 0,
    "generated_at": "2026-05-27T09:00:00Z"
  }
}
```

### Business rules

| Rule | Description |
|---|---|
| BR-SUM-01 | User must have permission `summary:generate` |
| BR-SUM-02 | Patient must have at least one clinical document or structured record |
| BR-SUM-03 | Generated summary status must be `draft` |
| BR-SUM-04 | Every generated summary must create `model_run` record |
| BR-SUM-05 | Every generated summary must create audit event `generate_summary` |
| BR-SUM-06 | If citation coverage below threshold, summary must be flagged for review |

# 20. Async Job Design

Summary generation có thể chạy sync hoặc async.

## 20.1 Sync option

Dùng cho MVP nhỏ:

```http
POST /api/v1/patients/{patient_id}/summaries/generate
```

Trả về summary khi đã tạo xong.

## 20.2 Async option

Dùng cho production hoặc summary dài:

```http
POST /api/v1/patients/{patient_id}/summaries/jobs
```

### Response

```json
{
  "success": true,
  "data": {
    "job_id": "job_001",
    "status": "queued"
  }
}
```

Check status:

```http
GET /api/v1/jobs/{job_id}
```

# 21. API Security Requirements

| Requirement | Description |
|---|---|
| API authentication | All endpoints except login require token |
| RBAC | Check permission before processing request |
| Patient-level access control | User can only access permitted patients |
| Audit middleware | Log sensitive actions automatically |
| Rate limit | Limit repeated generation and login attempts |
| Payload validation | Validate all request bodies |
| Sensitive data masking | Mask PHI in logs |
| TLS | All production API calls must use HTTPS |
| Idempotency key | Import and generation requests may support idempotency key |
| Error safety | Do not leak internal stack trace to client |
